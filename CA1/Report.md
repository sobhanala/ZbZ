# USART

## توضیح پیاده‌سازی USART

در این بخش ارتباط بین `ESP8266` و بورد `Usb` با دو مسیر جداگانه‌ی `TX/RX` پیاده‌سازی شده است. در کد پروتکل، داده‌ها در سطح بایت جابه‌جا می‌شوند و جزئیات فریم USART مثل `Start Bit` و `Stop Bit` توسط لایه‌های پایین‌تر پروژه مدیریت می‌شود؛ بنابراین تمرکز پیاده‌سازی روی خواندن از `rx`، نوشتن روی `tx` و نگه‌داری بافرهای ارسال و دریافت است.

---

## پیاده‌سازی USART روی ESP8266

مسیر فایل:

```text
include/CPS4042/Hardwares/Boards/Esp8266.h
```

در GPIO بورد، دو پایه‌ی زیر برای ارتباط سریال استفاده شده‌اند:

```cpp
Pins::Rx<WorkingVoltageTp> rx {BR, BTR, "Esp8266::rx"};
Pins::Tx<WorkingVoltageTp> tx {BR, BTR, "Esp8266::tx"};
```

پروتکل `usart` هم در سازنده‌ی بورد روی پردازنده نصب شده است:

```cpp
m_processor->installProtocol(&usart);
```

به این ترتیب، تابع `run()` این پروتکل در سیکل‌های پردازنده اجرا می‌شود و Sketch فقط با APIهای `read` و `write` کار می‌کند.

### تابع `write`

```cpp
void write(Byte byte) override
{
    m_outbound.push(byte);
}
```

این تابع بایت را مستقیم روی پین نمی‌نویسد، بلکه آن را در صف `m_outbound` قرار می‌دهد. ارسال واقعی در `run()` انجام می‌شود. این کار باعث می‌شود منطق Sketch از زمان‌بندی داخلی پروتکل جدا بماند.

### تابع `read`

```cpp
Byte read() override
{
    if(this->m_buffer.empty()) return 0;

    auto byte = this->m_buffer.front();
    this->m_buffer.pop();
    return byte;
}
```

داده‌هایی که از `rx` رسیده‌اند، قبلاً در `m_buffer` ذخیره شده‌اند. `read()` اولین بایت این بافر را برمی‌گرداند و از صف حذف می‌کند. در Sketch قبل از `read()` از `isDataAvailable()` استفاده شده، پس مقدار `0` فقط نقش حالت محافظتی برای بافر خالی را دارد.

### تابع `run`

```cpp
void run(Gpio& gpio) override
{
    gpio.tx.setCanRead(false);

    while(gpio.rx.hasByteToRead())
    {
        this->m_buffer.push(gpio.rx.read());
    }

    while(!m_outbound.empty())
    {
        gpio.tx.write(m_outbound.front());
        m_outbound.pop();
    }
}
```

اولین خط، خواندن از `tx` را غیرفعال می‌کند. این نکته برای جلوگیری از loopback ناخواسته لازم است؛ چون `tx` در این سمت فقط خروجی است و نباید داده‌ی ارسال‌شده‌ی خود بورد دوباره به عنوان ورودی خوانده شود.

در حلقه‌ی اول، هر بایت آماده از `rx` خوانده و وارد بافر پروتکل می‌شود. استفاده از `while` باعث می‌شود اگر چند بایت آماده باشد، همه‌ی آن‌ها در همان اجرای `run()` منتقل شوند.

در حلقه‌ی دوم، بایت‌های صف `m_outbound` روی `tx` نوشته می‌شوند. بعد از هر ارسال، همان بایت از صف حذف می‌شود تا دوباره ارسال نشود.

## پیاده‌سازی بورد Usb

مسیر فایل:

```text
include/CPS4042/Hardwares/Comm/Usb.h
```

برای طرف دوم ارتباط، بورد `Usb` با چهار پایه تعریف شده است:

```cpp
Pins::Vdd<WorkingVoltageTp> vdd {BR, BTR, "Usb::vdd"};
Pins::Gnd<WorkingVoltageTp> gnd {BR, BTR, "Usb::gnd"};
Pins::Rx<WorkingVoltageTp>  rx {BR, BTR, "Usb::rx"};
Pins::Tx<WorkingVoltageTp>  tx {BR, BTR, "Usb::tx"};
```

تنظیمات کلاس مطابق صورت پروژه روی `NotSpecified` قرار گرفته است:

```cpp
class Usb : public Board<BaudRates::NotSpecified,
                         BitRates::same(BaudRates::NotSpecified),
                         Frequency::F320khz, UsbVoltage, UsbGpio>
```

فرکانس بورد مستقل است و پروتکل USART در سازنده نصب شده است:

```cpp
m_processor->installProtocol(&usart);
```

کلاس داخلی `USART` در `Usb` از همان منطق سمت ESP8266 استفاده می‌کند: `write()` بایت را در صف خروجی می‌گذارد، `read()` از بافر ورودی می‌خواند، و `run()` داده را از `rx` به `m_buffer` و از `m_outbound` به `tx` منتقل می‌کند.

نکته‌ی اصلی این است که در `run()` این بورد هم `gpio.tx.setCanRead(false)` صدا زده شده است. با این کار، `tx` بورد `Usb` فقط مسیر ارسال پاسخ به میکروکنترلر است و برای خواندن درخواست‌ها استفاده نمی‌شود. درخواست‌ها از `Usb.rx` خوانده می‌شوند.

## Sketch هارددیسک

مسیر فایل:

```text
include/CPS4042/Sketchs/HardDisk.h
```

در `setup()` فقط آماده شدن Sketch اعلام می‌شود:

```cpp
std::cout << "HardDisk setup completed." << std::endl;
```

منطق اصلی در `loop()` قرار دارد:

```cpp
while(node()->usart.isDataAvailable())
{
    auto address = node()->usart.read();

    auto it   = m_storage.find(address);
    auto data = static_cast<Byte>(0);
    if(it != m_storage.end()) data = it->second;

    node()->usart.write(data);
}
```

هر بایتی که از USART برسد، به عنوان آدرس خوانده می‌شود. سپس مقدار متناظر از `m_storage` برداشته می‌شود و به میکروکنترلر برگردانده می‌شود. اگر آدرس داخل جدول وجود نداشته باشد، مقدار `0` ارسال می‌شود.

`m_storage` با `std::unordered_map<Byte, Byte>` پیاده‌سازی شده است؛ یعنی کلید، آدرس یک‌بایتی و مقدار، داده‌ی ذخیره‌شده در آن آدرس است. این برای نیاز این بخش کافی است، چون صورت پروژه فقط خواندن داده‌ی متناظر با یک آدرس بین `0` تا `255` را می‌خواهد.

در انتهای حلقه هم `delay(1)` قرار داده شده تا اجرای Sketch پشت سر هم و بدون فاصله نباشد.

## Sketch میکروکنترلر

مسیر فایل:

```text
include/CPS4042/Sketchs/Microcontroller.h
```

در `loop()` سه متغیر `static` وضعیت درخواست فعلی را نگه می‌دارند:

```cpp
static UByte nextAddress {0};
static bool  waitingForResponse {false};
static UByte lastRequest {0};
```

`nextAddress` آدرس بعدی برای خواندن است. `waitingForResponse` اجازه نمی‌دهد قبل از دریافت پاسخ، درخواست بعدی ارسال شود. `lastRequest` هم برای چاپ پاسخ کنار آدرس متناظر استفاده می‌شود.

ارسال درخواست:

```cpp
if(!waitingForResponse)
{
    auto request = static_cast<Byte>(nextAddress);
    node()->usart.write(request);
    lastRequest = nextAddress;
    std::cout << "[USART] request address: 0x" << std::hex
              << static_cast<int>(lastRequest) << std::dec << std::endl;

    waitingForResponse = true;
}
```

وقتی پاسخ قبلی هنوز در انتظار نیست، آدرس بعدی از طریق USART ارسال می‌شود و برنامه وارد حالت انتظار می‌شود. این پرچم لازم است؛ وگرنه ممکن بود چند آدرس پشت سر هم ارسال شوند و تشخیص این‌که پاسخ دریافتی مربوط به کدام درخواست است سخت شود.

دریافت پاسخ:

```cpp
while(node()->usart.isDataAvailable())
{
    auto data = static_cast<UByte>(node()->usart.read());
    std::cout << "[USART] response address 0x" << std::hex
              << static_cast<int>(lastRequest) << " -> data 0x"
              << static_cast<int>(data) << std::dec << std::endl;

    nextAddress++;
    waitingForResponse = false;
}
```

بعد از دریافت پاسخ، داده چاپ می‌شود، آدرس بعدی یک واحد افزایش پیدا می‌کند و میکروکنترلر از حالت انتظار خارج می‌شود. چون `nextAddress` از نوع `UByte` است، محدوده‌ی آدرس‌دهی همان یک بایت است.

## اتصال‌ها در `main.cpp`

مسیر فایل:

```text
src/main.cpp
```

دو بورد ساخته شده‌اند:

```cpp
Boards::Esp8266 esp8266;
Sensors::Usb     usb;
```

برای USART، دو لینک جداگانه بین `tx` و `rx` استفاده شده است:

```cpp
esp8266.gpio().tx.attachLink(linkTxToRx);
esp8266.gpio().rx.attachLink(linkRxToTx);

usb.gpio().rx.attachLink(linkTxToRx);
usb.gpio().tx.attachLink(linkRxToTx);
```

بنابراین مسیرها به شکل زیر هستند:

```text
ESP8266.tx -> Usb.rx
Usb.tx     -> ESP8266.rx
```

این اتصال ضربدری مهم‌ترین بخش سیم‌کشی USART است. درخواست آدرس از مسیر اول به هارددیسک می‌رسد و پاسخ داده از مسیر دوم برمی‌گردد.

لینک‌های تغذیه و زمین هم بین دو بورد مشترک شده‌اند:

```cpp
esp8266.gpio().vdd1.attachLink(linkRed);
esp8266.gpio().gnd1.attachLink(linkBlack);
usb.gpio().vdd.attachLink(linkRed);
usb.gpio().gnd.attachLink(linkBlack);
```

در پایان، Sketchها روی بوردهای مربوطه ساخته و اجرا می‌شوند:

```cpp
MicroController micro(&esp8266);
HardDisk        hardDisk(&usb);

micro.start();
hardDisk.start();

return Application::exec();
```

## سوالات

**سؤال ۱: توضیح دهید چرا بیت های شروع و پایان با بیت های داده اشتباه گرفته نمیشود.**

بیت‌های شروع و پایان با داده اشتباه گرفته نمی‌شوند، چون گیرنده جایگاه آن‌ها را از روی ترتیب زمانی فریم تشخیص می‌دهد، نه صرفاً از روی مقدارشان. خط در حالت بیکار روی `1` است. وقتی خط از `1` به `0` می‌رود و گیرنده داخل فریم قبلی نیست، این تغییر به عنوان `Start Bit` تشخیص داده می‌شود. بعد از آن، گیرنده می‌داند دقیقاً چند بیت داده باید بخواند و هر بیت را در زمان مشخص خودش نمونه‌برداری می‌کند.

پس اگر داخل داده مقدار `0` دیده شود، Start جدید حساب نمی‌شود، چون گیرنده در آن لحظه وسط فریم است. همین‌طور `Stop Bit` فقط در جایگاه انتهای فریم بررسی می‌شود و باید مقدار `1` داشته باشد. اگر در زمان Stop مقدار مورد انتظار دیده نشود، گیرنده آن را خطای فریم در نظر می‌گیرد، نه بیت داده.

در نتیجه مرز فریم با حالت idle خط، گذار شروع فریم، تعداد ثابت بیت‌های داده و جایگاه زمانی Stop Bit مشخص می‌شود.

**سؤال ۲: باتوجه به نبود کلاک توضیح دهید این پروتکل چه طور دادهها را تشخیص میدهد.**

در این حالت، کلاک جداگانه‌ای روی سیم ارسال نمی‌شود، اما دو طرف روی `Baud Rate` مشترک توافق دارند. Baud Rate طول زمانی هر بیت را مشخص می‌کند. گیرنده تا قبل از شروع فریم منتظر گذار Start می‌ماند؛ بعد از تشخیص Start Bit، زمان‌بندی داخلی خودش را با همان لحظه تنظیم می‌کند.

برای خواندن داده، گیرنده معمولاً کمی صبر می‌کند تا به مرکز بازه‌ی بیت برسد و سپس با فاصله‌های زمانی برابر، بیت‌های بعدی را نمونه‌برداری می‌کند. نمونه‌برداری از مرکز بیت باعث می‌شود خطاهای کوچک در لبه‌ها یا نویزهای لحظه‌ای کمتر اثر بگذارند.

بنابراین تشخیص داده در USART بر اساس کلاک مشترک انجام نمی‌شود؛ بر اساس Baud Rate مشترک، تشخیص Start Bit و نمونه‌برداری زمان‌بندی‌شده انجام می‌شود. اگر کلاک داخلی دو طرف یا Baud Rate آن‌ها اختلاف زیادی داشته باشد، نمونه‌برداری از مرکز بیت جابه‌جا می‌شود و احتمال خطای فریم بالا می‌رود.

**سؤال ۳: توضیح دهید آیا امکان پیادهسازی پروتکل  USART  به صورت نرمافزاری (با استفاده از پین های دیجیتال) در میکروکنترلرهایی که مجهر به سختافزار مربوطه نیستند امکان پذیر است؟ (به عنوان مثال این امکان برای  I2C وجود دارد) دلایل خود را کامل شرح دهید.**

بله، از نظر فنی امکان‌پذیر است و معمولاً با عنوان `Software UART` یا `Bit-Banging` شناخته می‌شود. در این روش برنامه خودش روی پین دیجیتال `Start Bit`، بیت‌های داده و `Stop Bit` را تولید می‌کند و در سمت دریافت هم با زمان‌بندی مشخص پین را نمونه‌برداری می‌کند.

اما سختی آن از I2C بیشتر است. در I2C خط کلاک وجود دارد و نرم‌افزار می‌تواند با کنترل `SCL` سرعت انتقال را کم یا زیاد کند. اگر پردازنده کمی دیرتر عمل کند، معمولاً می‌توان همان کلاک را دیرتر تغییر داد. ولی در USART کلاک جداگانه‌ای وجود ندارد و گیرنده باید در زمان درست نمونه‌برداری کند. تأخیر ناشی از وقفه‌ها، اجرای کدهای دیگر یا بار زیاد پردازنده می‌تواند باعث شود نمونه‌برداری از وسط بیت خارج شود.

به همین دلیل USART نرم‌افزاری برای Baud Rateهای پایین و برنامه‌های ساده قابل استفاده است، ولی برای ارتباط پایدار، نرخ‌های بالا یا Full Duplex واقعی مناسب نیست مگر این‌که زمان‌بندی خیلی دقیق کنترل شود. سخت‌افزار UART همین کارها را مستقل از CPU انجام می‌دهد: تشخیص Start، نمونه‌برداری بیت‌ها، بررسی Stop، بافر کردن داده و گزارش خطاهای فریم. بنابراین پیاده‌سازی نرم‌افزاری ممکن است، اما محدودتر و حساس‌تر از استفاده از سخت‌افزار USART/UART است.
